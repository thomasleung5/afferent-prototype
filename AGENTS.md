# Highest-priority rules

- This is a Vite + React + TanStack Router SPA with a Hono API server. It is not Next.js.
- Never introduce `next/link`, `next/navigation`, `app/` routing, server components, or `"use client"` directives.
- Work only on the requested task.
- Do not refactor unrelated files.
- Do not change UI patterns unless explicitly asked.
- Use existing components and CSS tokens.
- Do not edit generated files.
- Do not weaken auth, persistence, CORS, origin checks, or production safety gates.
- Never log secrets, bearer tokens, uploaded file contents, request bodies, recovery-token query strings, or PII.
- Always run the nearest relevant tests before declaring work complete.

# Architecture

This app uses:

- Vite + React + TypeScript
- TanStack Router with file-based routes
- Zustand for the build-model store
- Hono for API routes
- Supabase Auth for real user authentication
- Supabase-backed persistence for server-side studies
- Plain CSS with design tokens

Important paths:

- `src/main.tsx` mounts `<RouterProvider />`.
- `src/routes/` contains file-based routes.
- `src/pages/` contains page bodies.
- `src/routeTree.gen.ts` is generated by TanStack Router and must not be edited by hand.
- `lib/store.ts` owns the Zustand build-model state.
- `server/` contains the Hono API.
- `components/ui/`, `components/layout/`, and `components/table/` contain reusable UI.
- `features/` contains screen-level feature components.
- `styles/tokens.css` and `src/index.css` define design tokens.

The path alias `@/*` resolves to the project root.

# Styling and UI consistency

Plain CSS only.

- Use existing CSS variables and design tokens from `src/index.css` and `styles/tokens.css`.
- Do not introduce Tailwind, CSS-in-JS, `styled-components`, `emotion`, or another styling system.
- Do not hardcode colors, font sizes, spacing, borders, shadows, or radii.
- Do not create one-off button, card, table, modal, badge, or layout styles when an existing component exists.
- Use existing components from:
  - `components/ui/`
  - `components/layout/`
  - `components/table/`
- Inline style objects using tokens are acceptable for one-off layouts when consistent with existing patterns.
- Recurring UI belongs in shared components.
- If a new reusable UI pattern is needed, add it to the appropriate shared component folder and explain why.

Claude Review should flag:

- Duplicate UI components
- One-off visual styles
- Unrequested spacing, color, typography, or layout changes
- Hardcoded visual constants
- Unrelated visual changes

# Generated files

- Do not edit `src/routeTree.gen.ts` by hand.
- It is generated by the TanStack Router Vite plugin.
- Running `npm run build` or `npm run dev` regenerates it.
- Do not commit generated artifacts unless the repo explicitly tracks them.

# Auth and security

- Never trust client-side auth for authorization.
- Server routes must enforce user/org authorization.
- Do not weaken Supabase JWT verification.
- Do not weaken CORS, origin allowlist, production boot checks, or environment validation.
- `AUTH_DEV_BYPASS=1` is development-only and must never affect production behavior.
- Do not expose or log:
  - `SUPABASE_SERVICE_ROLE_KEY`
  - bearer tokens
  - authorization headers
  - uploaded PDF/Excel contents
  - request bodies
  - recovery-token query strings
  - user emails or PII
- Error responses should follow the project convention and avoid leaking stack traces.

Flag any PR that risks:

- Auth bypass
- Cross-org data access
- Client-side-only authorization
- Token leakage
- Logging sensitive data
- Production safety gates being weakened

# Persistence and data safety

The app has two coordinated persistence layers:

- Zustand/localStorage editing state
- Server-side study persistence via `/api/studies/*`

Rules:

- Do not remove the localStorage fallback.
- Do not overwrite local edits unexpectedly.
- Do not change autosave, snapshot, restore, destructive actions, or active-study behavior without tests.
- Preserve the distinction between local-only editing and server-backed studies.
- Loading server snapshots must avoid autosave ricochet behavior.
- Demo/sandbox workspace changes must not accidentally save into a real active server study.
- Any change to `/api/studies/*`, Supabase migrations, `lib/store.ts`, autosave, snapshots, restore, or active-study behavior requires tests or a clear explanation if not testable.

Flag any PR that risks:

- Data loss
- Cross-user or cross-org study access
- Saving demo data into a real study
- Overwriting active studies unintentionally
- Breaking localStorage fallback
- Breaking version/snapshot restore behavior

# Import/export correctness

Preserve the existing separation:

- AI-backed PDF parsing lives under `/api/ai/*`.
- Deterministic Excel preview lives under `/api/import/*`.
- Server Excel preview is side-effect-free.
- Client-side domain converters merge parsed data into the Zustand store.

Rules:

- Do not mix AI parsing with deterministic Excel preview logic.
- Preserve normalized domain schemas for:
  - services
  - direct labor
  - operating expenditures
  - volume
  - fees
  - cost allocation / overhead
- API responses should follow:
  - `{ ok: true, ... }`
  - `{ ok: false, message }`
- Import changes should include fixture tests where possible.
- Export changes should preserve PDF/Excel formatting and include regression coverage where possible.

Flag any PR that risks:

- Silent import data corruption
- Incorrect fee, labor, volume, or operating normalization
- Broken PDF or Excel export
- Server-side writes from preview endpoints
- AI parsing being used where deterministic parsing is expected

- Tokens

# Verification

Run the nearest relevant checks before completion.

Use targeted checks for focused changes:

- TypeScript: `npm run typecheck`
- Full test suite: `npm test`
- Browser smoke tests: `npm run test:smoke`
- First-time Playwright setup: `npm run test:smoke:install`

Suggested verification by change type:

- UI/routing changes:
  - `npm run typecheck`
  - `npm run build`
  - `npm run test:smoke`

- Server/API changes:
  - `npm run typecheck`
  - relevant `npm run test:<name>`
  - `npm test`

- Auth/persistence changes:
  - `npm run typecheck`
  - `npm run test:auth`
  - persistence-related tests
  - `npm test`

- Import/export changes:
  - relevant parser/export tests
  - `npm test`

- Styling-only changes:
  - `npm run typecheck`
  - `npm run build`
  - smoke test if navigation, layout, or critical flows are affected

If a command cannot be run, state exactly why.

# Local development notes

- `npm run dev` launches Vite on `:3000` and Hono on `:8787`.
- Vite proxies `/api/*` to `:8787`.
- Local API development usually needs `.env.local`.
- `AUTH_DEV_BYPASS=1` may be used only in development.
- AI parsing requires `ANTHROPIC_API_KEY`.
- Production requires Supabase env vars and `ALLOWED_ORIGINS`.
